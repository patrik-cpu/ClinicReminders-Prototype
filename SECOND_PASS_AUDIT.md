## 1. Executive Summary

No active local P0 was found in this second pass. The current tree compiles, dependency consistency passes, CI-pattern tests pass, full unittest discovery passes, and the local pilot release script passes while skipping live Google checks because no service-account credentials are available locally.

The repository is materially safer than the original reports describe. Remember-login query tokens, Authlib pinning, upload resource limits, unsafe HTML hot spots, weak new-password paths, profile rename failure handling, delete ordering, timezone handling, stats tab labeling, and several stats calculation/display issues have been improved and now have regression coverage.

The remaining highest risks are still structural:

- Google Sheets and Drive are shared multi-tenant backend resources, with tenant isolation enforced mostly by application code.
- `reminders_app_v3.py` is now 16,312 lines and still mixes UI, auth, Google I/O, upload ETL, reminder generation, outcome analytics, CSS, and dormant code.
- Multi-step Google mutations are not transactional.
- There is no browser-level E2E suite and no credentialed live Google smoke in CI.
- External Google calls remain synchronous and mostly lack operation-level timeouts.

One new P1 user-visible logic issue was found in current outcome matching: multiple sent reminder steps for the same item purchase cycle are deduped before success matching, and the dedupe keeps the earliest sent step. This can ignore a later overdue/reminder step that should make the "success window after sent date" count as a success.

## 5. New Issues Found In This Pass

### P1: Later sent reminder steps are ignored by post-reminder success matching

Evidence:

- `build_reminder_outcomes` reduces sent actions by calling `dedupe_outcome_sent_records` before matching: `reminders_app_v3.py:12798`.
- `dedupe_outcome_sent_records` keeps the earliest sent record per purchase cycle: `reminders_app_v3.py:12437`.
- post-reminder success then only sees that retained sent date: `reminders_app_v3.py:13056`.

Reproduced locally with this scenario:

- Rabies vaccine billed on 2024-05-01.
- Due date 2025-05-01.
- First reminder sent 2025-04-01.
- Overdue/later reminder sent 2025-05-20.
- Repeat purchase on 2025-05-24.
- Due-date window is 14 days; post-reminder window is 7 days.

Actual current result: one deduped outcome row with `Sent Date = 2025-04-01` and `Outcome = No Match`.

Expected product behavior: one outcome row for the purchase cycle, but success should be counted because the 2025-05-24 purchase is within 7 days of the 2025-05-20 sent reminder.

Suggested fix:

- Keep one outcome row per purchase cycle, but preserve all sent dates for that cycle during matching.
- Compute success candidates from every sent step, then collapse to one success/no-match/pending outcome per cycle.
- Add a regression test for the exact scenario above and a grouped Rabies+Tricat version.

Safe to fix now: yes, as a focused stats/outcome PR with regression tests.

### P2: Some now-removed stats concepts still have inactive helpers

Evidence:

- `statistics_summary_for_period` is only referenced by tests: `reminders_app_v3.py:11952`.
- `build_statistics_daily_frame` has no active references: `reminders_app_v3.py:11987`.

Why it matters:

This looks like residue from the old Statistics overview/daily tabs. It is not currently a user-facing bug, but it adds confusion and may keep stale semantics alive in tests.

Suggested fix:

- Do not delete immediately. First confirm the old overview/daily views are intentionally gone, then remove or quarantine the inactive helpers and obsolete tests in one small cleanup PR.

### P2: Outcome revenue formulas need domain confirmation after recent additions

Evidence:

- `Revenue per Year`, `Theoretical Max Revenue`, `Capturable Revenue per Year`, and `Captured Revenue %` are computed in `build_reminder_outcomes`: `reminders_app_v3.py:12936`.

Why it matters:

The current theoretical max uses the faster of actual and desired cadence (`np.fmax(actual_interval_values, desired_interval_values)`). That fixed earlier over-100% captured rates, but the product definition should be documented in code/tests because this is business-facing financial reporting.

Suggested fix:

- Add explicit tests for actual gap less than desired, actual gap greater than desired, zero/unknown gap, and zero revenue.
- Add tooltip/docs copy stating the formula.
